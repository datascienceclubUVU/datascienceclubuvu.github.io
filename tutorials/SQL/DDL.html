SQL Tutorials: DDL

DISCLAIMER: Before starting this tutorial, it's highly recommended that you download an RDBMS to follow along. We recommend using either PostgreSQL, MS SQL Server, or MySQL.

Now that you have a clear understanding of the core concepts behind Relational Databases, let's learn how to create, update, and delete database objects. Data Definition Language (DDL) is a form of SQL used specifically for creating, altering, and dropping database objects. Listed below are the learning objectives in this lesson:

  • Creating Database Objects
  • Altering Database Objects
  • Dropping Database Objects
  • Truncating Database Objects
  • Constraints

Creating Database Objects

As mentioned in the previous tutorial, Relational Databases are comprised of four primary objects: Databases, Schemas, Tables, and Attributes. The videos below describe how to create the first three of these objects:

    How to Create A Database
    How to Create A Schema
    How to Create A Table

Summary:

  • The CREATE DATABASE statement is used to create databases in an RDBMS. Once you have multiple databases created, it's helpful to use the optional IF NOT EXISTS keyword to check if the name has already been used.
  • The CREATE SCHEMA statement is used to create a schema within a database. These can be used to differentiate different business units and provide security for sensitive data. It's recommended to use this this statement in conjunction with the USE DATABASE [db_name] statement to ensure you create the schema in the correct database.
  • The CREATE TABLE statement is used to define the name and structure of a table in a database. You must specify each column and its associated data type in the table. It's recommended to use this statement in conjunction with the USE DATABASE [db_name] statement to ensure you create the table in the correct database.

Altering Database Objects

Sometimes, you may need to rename or restructure a database obejct. View the following views to learn more about these options:

    How to Alter Database Tables
    How to Rename Databases

Summary:

  • To add a column/attribute to a table, use the following syntax:
    ALTER TABLE [table_name] ADD [col_name] [data_type];
  • To drop a column/attribute from a table, use the following syntax:
    ALTER TABLE [table_name] DROP [col_name];
  • To change an attribute's data type, use the following syntax (differs between RDBMSs):
    ALTER TABLE [table_name] MODIFY [col_name] [new_data_type] [constraint]
  • To Rename a database, use the following syntax (differs between RDBMSs):
    ALTER DATABASE [db_name] MODIFY name = [new_name];

Dropping Database Objects

Just like in any programming language, there is more than one type of data. This is because some data may be numeric in nature, whereas other might be textual in nature. Let's take a look at the various data types:

    Common SQL Data Types

Summary:

  • VARCHAR/CHAR data types represent textual data. These are short for VARying CHARacter and CHARacter, respecitively. The primary difference between these two is that CHAR is a fixed limit on the number of characters allowed, whereas VARCHAR allows a strong of characters up to a specified limit.
  • DECIMAL/NUMERIC/FLOAT data types represent non-integer numerical data. These are used interchangeably and represent numbers with a fixed number of decimal points.
  • INT/INTEGER data types represent numerical data without deciaml places. These are used interchangeably and are used to represent attributes with discrete values (numbers that can't be partial).
  • DATE/TIME/DATETIME data types represent time-series data, or data that occurs along a timeline. This may or may not include timezone information or may simply be limited to just a date.

Truncating Database Objects

Before you get started with designing your first Relational Database, it's critical to understand the different types of relationships available. Let's review them down below:

    Types of Table Relationships
    How to Manage Many-to-Many Relationships

Summary:

  • One-to-Many Relationships are the most common relationship and are represented by a lookup table representeing distinct items in the database and another table that references the items in that table multiple times via a foreign key.
  • Many-to-Many Relationships should be avoided whenever possible. These are represented by two tables that have multiple entries in one table that match with multiple entries in the other table. These should be avoided because it makes understanding business logic much more difficult.
