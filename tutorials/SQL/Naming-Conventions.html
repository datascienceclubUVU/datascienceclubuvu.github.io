SQL Tutorials: Naming Conventions

DISCLAIMER: Before starting this tutorial, it's highly recommended that you download an RDBMS to follow along. We recommend using either PostgreSQL, MS SQL Server, or MySQL.

Welcome to the last Beginner-level tutorial! To finish off the basics, we will learn how to write clean SQL and best practices about formatting code. In this tutorial, we will cover the following learning objectives:

  • SQL Capitalization Notation
  • Snake Case vs. Camel Case
  • Object Naming Best Practices
  • Alias Naming Best Practices

SQL Capitalization Notation

As you may have noticed over the past several tutorials, it's common to capitalize keywords and clauses in queries. Why? Simply because it vastly improves readability. If you are selecting ten columns from a table, how do you know where the "from" clause starts? Read the following article to get a better understanding of this concept:

    SQL Syntax Conventions

Summary

  • "All SQL commands should be written in uppercase letter. This allows the reader to identify the keywords in the SQL statement and easily determine what the query is executing." (dataschool.com, 2021). Example:
    SELECT first_name, last_name, street, city, state FROM customers WHERE city = 'Los Angeles';
  • "Lay out each column or logical statement on new lines so that it['s] easy to identify what is being queried and filtered." (dataschool.com, 2021). Example:
    SELECT first_name, last_name, street, city, state
    FROM customers
    WHERE city = 'Los Angeles';
  • "[When grouping,] It is generally recommended to use column names [rather than column index numbers] to be clear as to what you are grouping by" (dataschool.com, 2021). Example:
    SELECT first_name, COUNT(*) AS num_names
    FROM customers
    GROUP BY first_name;

Snake Case vs. Camel Case

If you're a seasoned programmer, you understand the difference between Snake Case (snake_case) and Camel/Pascal Case (camelCase). These both have their use cases, but both also have their places. Read the following article (Snake Case Section) to get a better understanding of this concept:

    Database, Table, and Column Naming Conventions

Summary:

  • Snake Case (snake_case) is best for naming primary and foreign keys.
  • Camel Case (camelCase) is best for naming columns and tables.

Object Naming Best Practices

One of the first things you learn when programming is to name your variables something meaningful. For example, if you have a variable representing an animal's name, rather than naming the variable "x" or "myVar", name it something like "animal_name". This same principle applies when naming database objects. Read the following article to get a better understanding of this concept:

    Object Naming Best Practices

Summary:

  • Tables: "If you're naming entities that represent real-world facts, you should use nouns... If possible, use a single word that exactly describes what is in the table." (sqlshack.com, 2020):
    CREATE TABLE customers ...;
  • Primary Keys: "You should usually have only 1 column serving as a primary key. It would be the best to simply name this column "id". You should also name your PK constraint in a meaningful way." (sqlshack.com, 2020):
    ALTER TABLE customers ADD CONSTRAINT customers_pk PRIMARY KEY (customer_id);
  • Foreign Keys: "[With foreign keys], since they store values from the range of primary key of the referenced table, you should use that table name and "id"." (sqlshack.com, 2020):
    ALTER TABLE orders ADD CONSTRAINT customers_fk FOREIGN KEY (customer_id) REFERENCES customers (customer_id);
  • Data Columns: "You should use the least possible words to describe what is stored in that column." (sqlshack.com, 2020):
    ALTER TABLE customers ADD date_joined DATE;

Alias Naming Best Practices

As mentioned in the previous tutorial, it's a best practice to give long column names or aggregated columns aliases using the "AS" keyword. These can also be used to give temporary names for tables when doing joins. Watch the following video to get a better understanding of this concept:

    SQL Aliases Best Practices

Summary:

  • "Column Aliases: [These] are used to make column heads in your result set easier to read." (techonthenet.com, 2022). Example:
    SELECT first_name, date_first_joined AS join_date
    FROM customers;
  • "Table Aliases: are used to shorten your SQL to make it easier to read or when you are performing a self join." (techonthenet.com, 2022). Example:
    SELECT customer_id, first_name, last_name
    FROM customers AS c
    JOIN orders AS o ON c.customer_id = o.customer_id;
